编译环境配置

编译环境配置

boi支持为不同的环境配置对应的编译选项。boi将环境划分为三种：development、testing和production，三者的定义与执行时机如下表：

名称 对应配置项 说明 执行时机 uglify是否可用 sourcemap 备注
development dev 本地开发环境 boi serve N Y entry包括hmr依赖模块
testing testing 测试环境 boi build 和boi build testing Y N --
production prod 生产环境 boi build prod Y N --
  • development是本地开发环境，只在开启dev server时生效，所以针对development环境的配置项不会影响最终的编译结果；
  • testing是测试环境，编译结果可用于上线之前的内部测试；
  • production是生产环境，运行在线上，直接面对用户。

boi之所以将以上三种环境进行区分，是为了覆盖一个前端项目从始至终的任何一个阶段。development环境下，boi提供本地服务容器和mock服务，方便FE不依赖后端接口进行独立开发；而区分testing环境和production环境是由于前端资源都是静态的，代码写死了就不会根据运行环境改变。最常见的场景是：

  • js脚本文件与主站是不同域的，进行异步请求时需要完整的url而不是path；
  • 测试环境与生产环境的主站域名不相同；
  • 测试完毕之后，如果需要提交上线，必须修改js脚本中接口的url。

这种场景下麻烦的是最后一步，要么人工手动修改，要么通过工具自动修改。即使使用工具，也需要手动修改工具的配置项。所以不论哪一种方式，都存在不必要的手动操作，不仅麻烦，而且容易出错。

boi对以上场景的解决方案是：结合环境配置和JavaScript的define功能，为不同环境配置对应的配置项。比如：

boi.spec('js',{
  dev: {
    useHash: false,
    define: {
      LOGIN_API: '/login'
    }
  },
  testing: {
    uglify: false,
    useHash: true,
    define: {
      LOGIN_API: '//passport.djtest.com/login'
    }
  },
  prod: {
    uglify: true,
    useHash: true,
    define: {
      LOGIN_API: '//passport.daojia.com/login'
    }
  }
});

boi.mock({
  'GET /login': {
    ok: {
      code: 100,
      msg: '登录成功',
      data: {
        user: 'boi'
      }
    },
    fail: {
      code: 200,
      msg: '登录失败'
    }
  }
});

以上配置最终的表现效果为：

  • development环境下将登录api映射到本地，然后配置本地mock服务器返回假数据；
  • 执行boi build testing或者boi build（默认testing环境），根据testing环境的配置项，编译输出的文件中登录api被替换为测试url，并且代码不压缩；
  • 执行boi build prod，根据production环境的配置项，编译输出的文件中登录api被替换为线上url，并且代码经过压缩混淆。

目前分环境配置功能只支持通过boi.specAPI进行配置。
